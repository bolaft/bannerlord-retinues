# Build instructions

## Prerequisites

- **.NET SDK 8.0+** - `dotnet --version` should show 8.x or newer.
- **Python 3.8+** - used by `tpl/prefabs.py`, `loc/strings.py` and `build.py`.
- **Git Bash/WSL** (recommended on Windows) to run `build.sh` as provided.
- **7‑Zip** (`7z`) - optional but required to create release archives when packaging for Workshop.
- Read access to your Bannerlord installation directory (e.g.  
  `C:\Program Files (x86)\Steam\steamapps\common\Mount & Blade II Bannerlord`).

> Game assemblies target **.NET Framework 4.7.2**. You do not need to install that runtime; the project references the game's DLLs from `dll/12` or `dll/13`.

---

## One‑time setup

1) **Clone** the repository.

2) **Provide a local override for your Bannerlord install path** (optional but recommended).

MSBuild picks up `Directory.Local.props` if present; `build.sh` also looks for `Retinues.Local.props` when deciding the game directory.

Example `Directory.Local.props` (used by MSBuild):

```xml
<!-- Directory.Local.props -->
<Project>
  <PropertyGroup>
    <BannerlordGameDir>C:\Program Files (x86)\Steam\steamapps\common\Mount &amp; Blade II Bannerlord</BannerlordGameDir>
    <DefaultConfiguration>Debug</DefaultConfiguration>
  </PropertyGroup>
</Project>
```

Example `Retinues.Local.props` (optional; read by `build.sh` to determine game dir):

```xml
<Project>
  <PropertyGroup>
    <BannerlordGameDir>C:\Program Files (x86)\Steam\steamapps\common\Mount &amp; Blade II Bannerlord</BannerlordGameDir>
  </PropertyGroup>
</Project>
```

- Keep these files at the repo root and do not commit them (they are git‑ignored).
- Alternatively set environment variable `BANNERLORD_GAME_DIR` before running `build.sh` to override the path.

3) Populate the `dll` cache with game assemblies for the Bannerlord versions you intend to build for:

```
dll/12/**   # BL 1.2 DLL set
dll/13/**   # BL 1.3 DLL set
```
The build selects the appropriate set with `-v`/`--version` (default is 13).

4) Place required third‑party DLLs into both `dll/12` and `dll/13` as needed by the module:
- `0Harmony.dll`
- `Bannerlord.UIExtenderEx.dll`

## Project layout

```
build.sh                               # Build wrapper (requires bash)
build.py                               # Packaging / SubModule.xml generator
build.yaml                             # Build configuration (versions, workshop, paths)
Directory.Build.props                  # Central MSBuild defaults
Directory.Build.targets                # Deploy & staging targets
Directory.Local.props                  # (optional) MSBuild local overrides

xml/Retinues/**                        # -> ModuleData/** deployed to module
loc/Retinues/Languages/**              # -> ModuleData/Languages/**
tpl/**                                 # GUI template tooling (prefabs)

src/Retinues/Retinues.csproj           # The game module project
gui/Retinues/                           # Generated GUI prefabs land here
out/                                    # Build outputs (see notes below)
```

**Artifacts**
- Raw project outputs: `out/bin/<Project>/<Configuration>/...`
- Prefabs output: `gui/<ModuleName>/` (generated by `tpl/prefabs.py`)
- Deployed module folder: `<BannerlordGameDir>/Modules/<ModuleName>/`
- Release packaging and Workshop XMLs: `Releases/<BL>/<ModuleName>/`

> Deploy targets will remove existing files in the deployed module folder except `*.log` files.

## Quick builds and options

The provided `build.sh` is a convenience wrapper that runs prefab generation, string compilation, `dotnet build`, deploy, and (optionally) release packaging via `build.py`.

Common flags recognized by `build.sh`:
- `-v, --version <12|13>` : Bannerlord version key (default: 13)
- `-r, --release <N>`     : Build Release and override the last version segment with `<N>` (triggers packaging steps)
- `--prefabs`             : Run only prefabs generation
- `--no-prefabs`          : Skip prefabs generation
- `--strings`             : Run only `loc/strings.py` (string compilation)
- `--no-strings`          : Skip `loc/strings.py`
- `--no-deploy`           : Do not copy files into the game Modules directory

Examples

### Development build (Debug) and deploy to game (default behavior)
```bash
./build.sh -v 12
```
- Runs prefab generation, string compilation, `dotnet build` (Debug), and deploys outputs and generated `SubModule.xml` into your game Modules folder.
- Copies `UIExtenderDebug.xml` to the deployed module root when building Debug.

### Prefabs only
```bash
./build.sh --prefabs
```
- Runs `tpl/prefabs.py` to render GUI prefabs into `gui/<ModuleName>/`.
- If `--prefabs` and `--no-deploy` are combined, prefabs are not copied into the game folder.

### Strings only
```bash
./build.sh --strings
```
- Runs `loc/strings.py --module <Module>` to compile localization into `ModuleData/Languages/`.

### Release build with version bump and packaging
```bash
./build.sh -r 8
```
- Builds `Release` (no `UIExtenderDebug.xml`), deploys to the game module directory (unless `--no-deploy`), and when deploy is enabled will:
  - write `SubModule.xml` with the patched version (last segment set to `8`),
  - copy the deployed module into `Releases/<BL>/<ModuleName>/`,
  - emit `WorkshopCreate<BL>.xml` / `WorkshopUpdate<BL>.xml` into `<Game>/bin/Win64_Shipping_Client/`,
  - create a `Retinues_v<fullVersion>.zip` using `7z` (requires `7z` on PATH or installed in common locations).

### Other useful flags
- `--no-deploy` prevents copying files into your game folder (useful for CI or when packaging only).

## Packaging and release notes

- `build.py` reads `build.yaml` and can either write `SubModule.xml` (`--out <module_dir>`) or prepare a release package and Workshop XMLs (`--package-release --module-dir <game_module_dir>`).
- `build.py` will try to find `7z` (first on PATH, then common Windows install paths). If `7z` isn't found, packaging will skip archive creation but will still produce the Workshop XMLs and copy the `Releases/<BL>/<ModuleName>/` folder.
